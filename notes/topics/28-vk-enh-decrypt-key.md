# `vk::enh::decrypt` — статический ключ в `libEnhancementLibShared.so`

В [[15-on-device-asr-kws-diarization]] был замечен экспортированный символ:

```
vk::enh::decrypt(std::vector<unsigned char>)
```

Декомпиляция через `rizin pdg @ 0x130360` показывает, что функция:

1. Конструирует временный объект (вероятно, контекст крипто-движка) через `fcn.0055c2c0(&ctx, 0x83513, 0x83523)` — то есть **передаёт два указателя в `.rodata`** на статические байтовые блобы.
2. Вызывает `fcn.0055c868(&ctx, data_ptr, data_len)` — обрабатывает входной вектор.
3. Возвращает результат.

## Что лежит в `.rodata` по этим адресам

`px 32 @ 0x83513`:

```
0x00083513:  6f 8c c4 b7 19 d4 0d 16  d1 fc b9 ba bb c4 7d 7e   ← 16 байт
0x00083523:  27 1f 82 20 bf 05 37 56  5f bb 20 01 f8 9f d5 a6   ← ещё 16 байт
```

Размер каждого блока — ровно 16 байт. По расположению пары и смещения `+0x10` между ними это **похоже на AES-128 key + IV** (или, что то же самое, на два ключа для какого-то stream-cipher с двумя слотами).

Функции `fcn.0055c2c0` и `fcn.0055c868` без отдельной декомпиляции не классифицируются однозначно (могут быть OpenSSL `EVP_CipherInit_ex`/`EVP_CipherUpdate`-обёртками, либо собственным AES-движком на Boost).

## Что это значит

`vk::enh::decrypt(...)` принимает байтовый вектор и расшифровывает его статическим, вкомпиленным в `libEnhancementLibShared.so` ключом. Это значит:

- Любой, у кого есть APK MAX 26.15.3, **может расшифровать любой артефакт, обработанный этой функцией**, имея только её исходный массив байт.
- Это не «security» — это «light obfuscation», известный паттерн для хранения внутренних blob-ов (моделей, конфигов) в зашифрованном виде, чтобы их нельзя было просто выдернуть строкой `strings`.
- Скорее всего применяется для зашифрованных TFLite-моделей или конфигов, лежащих в `.rodata` той же `.so` (см. [[15-on-device-asr-kws-diarization]] §10), либо для проверки checksum-а скачанной с сервера модели в `KwsFeatureDelegate` (см. [[16-server-pushed-ml-models-in-calls]]).

## Чего не достаточно

- Алгоритм (AES-CBC, AES-CTR, ChaCha20, кастомный?) — без полной декомпиляции `fcn.0055c2c0` / `fcn.0055c868` сказать нельзя.
- К каким именно блобам это применяется — нужно найти call-site `vk::enh::decrypt` внутри `.so`. По беглому просмотру afl + axt ссылок на `decrypt` я не нашёл — это **экспортируемый API**, вызываемый снаружи (вероятно, factory классов через C-обёртки `c_enh_*`).

## Скептический итог

Это **не private key для дешифровки трафика клиента-сервера**. Это статический ключ для какого-то внутреннего blob-обфускатора в audio-enhancement стеке (vk::enh::*). По формату ключ+IV (16+16 байт) — похоже на AES-128, но требует добитой декомпиляции `fcn.0055c2c0`. С точки зрения «backdoor-индикатор» — **не indicator**, типичная light-obfuscation штука для embed-моделей.

Однако сам факт того, что в коде ML-стека есть обвязка `decrypt + dlopen` (см. [[15-on-device-asr-kws-diarization]] §9), а серверно-доставляемые модели (см. [[16-server-pushed-ml-models-in-calls]]) приходят как `*.tflite + *.cfg` — складывается в потенциальный путь «расшифровать blob → подгрузить как .so». Прямой улики такого использования в текущей сборке я не нашёл, но архитектурно это возможно, и стоит проверить отдельно.
